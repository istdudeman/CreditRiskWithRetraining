# Train Synthetic Test Real (TSTR) Methodology

This document outlines the step-by-step evaluation process of the TSTR Quality Gate program designed to evaluate the utility and privacy of generated synthetic credit risk data.

## 1. Data Split and Ingestion
The evaluation pipeline requires two distinct datasets to prevent data leakage:
- **Dsyn (Synthetic Data):** The dataset generated by a synthesizer (e.g., CTGANSynthesizer or VGM).
- **Dreal (Real Data):** A hold-out slice of the original public dataset that was never seen by the synthesizer during its training phase.

The program reads both datasets and further splits `Dreal` into an 70/30 train-test split (`Dreal_train` and `Dreal_test`). `Dreal_test` serves as the ultimate benchmark for all evaluations.

## 2. Privacy & Fidelity Checks
Before applying any transformations, the program computes privacy and distributional fidelity metrics against the continuous numerical columns:
- **Distance to Closest Record (DCR):** To ensure the generative model is not merely memorizing the training data, the program calculates the Euclidean distance from each synthetic record to its nearest neighbor in `Dreal`. A substantial fraction of records with a DCR of 0.0 indicates exact copying (data leakage).
- **Inverted KS Statistic (1 - KS):** The Kolmogorov-Smirnov (KS) statistic measures the maximum distance between the cumulative distribution functions (CDFs) of `Dsyn` and `Dreal`. We calculate the "Inverted" KS (1 - KS), where a value closer to 1.0 indicates higher fidelity (the distributions are highly similar).

## 3. Preprocessing (WoE Transformation)
To ensure that categorical and continuous features are treated identically by the evaluation model:
- The pipeline utilizes a custom `WoETransformer`.
- It dynamically bins continuous features (ignoring high-cardinality string IDs) and replaces categorical values with their Weight of Evidence (WoE) scores.
- Importantly, it fits one `WoETransformer` independently on `Dreal_train` for the TRTR baseline and a separate `WoETransformer` on `Dsyn` for the TSTR model. `Dreal_test` is transformed using the respective fitted mappers to prevent data leakage from the test set.

## 4. Model Definition & Monotonic Constraints
The evaluation uses XGBoost `XGBClassifier` models. To align with credit risk banking principles (where higher income or financial stability should denote lower risk):
- The script applies Monotonic Constraints (`-1` for decreasing relationship) to relevant features. 
- Because features are transformed into WoE mapped values (where typically higher WoE signifies lower default probability), these constraints help ensure the XGBoost model does not learn counter-intuitive relationships.

## 5. Model Execution (TRTR vs TSTR)
The program establishes a baseline and then compares the synthetic data to it:
- **TRTR Baseline (Train Real Test Real):** A model is trained on `Dreal_train` and evaluated against `Dreal_test`.
- **TSTR Evaluation (Train Synthetic Test Real):** A separate model is trained entirely on `Dsyn` and then evaluated against the same `Dreal_test`.

For both runs, the script calculates the Area Under the Receiver Operating Characteristic Curve (**AUC**) and the **F1-Score** (critical for evaluating the 14.8% default minority class imbalance).

## 6. MLOps Quality Gate
As a final step, the program acts as an automated Quality Gate in an MLOps lifecycle:
- It tracks all parameters and metrics (AUC, F1, DCR, KS) locally via an **MLflow** database natively created in the project folder (`mlruns/`).
- It compares the TSTR AUC against the TRTR AUC.
- If the utility of the synthetic data is poor (defined as the TSTR AUC dropping more than **10%** from the baseline TRTR AUC), the pipeline flags the run with a `"Low Utility"` metric, notifying downstream processes that the generated data is unsuitable for final production training.
